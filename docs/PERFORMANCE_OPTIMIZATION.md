# 性能优化思路

这份文档记录当前战斗与寻路优化的核心思路，方便后续继续做单位、卡牌、建筑和 AI 时不把性能问题重新写回来。

## 总原则

这种浏览器 RTS / 卡牌游戏的性能问题通常不是某个公式太贵，而是重复工作太多：

- 每帧全局遍历。
- O(N2) 单位互查。
- 目的地不变还反复寻路。
- 每次命中都创建 canvas、texture、geometry、material 或 DOM。
- Profiler 标签太粗，无法分辨真正耗时的是移动、命中、特效还是清理。

优化顺序：

1. 先确认高耗时行是父级汇总还是具体子项。
2. 把粗标签拆细。
3. 去掉重复工作。
4. 再考虑缓存、对象池和降频。
5. 最后才动玩法数值和视觉表现。

## 当前优化后的架构

- `UnitRegistry`：单位出生/死亡主动注册和注销，不靠每帧过滤死亡单位。
- `UnitLogicSystem`：单位自己的状态机入口，负责 idle、moving、chasing、attacking、knockback、dead。
- `TargetingSystem`：空间网格索敌，避免全场单位扫描。
- `PathfindingSystem`：独立 A* 寻路，支持 worker。
- `MovementAgent`：单位自己的路径、目的地、击退和移动状态。
- `AttackSystem`：攻击事件、投射物、弹体池和命中检测。
- `CombatSystem`：只负责伤害结算、闪避、护盾/生命扣除、击退、飘字和死亡触发。
- `EffectsSystem`：视觉效果和可复用纹理/特效缓存。

## 已处理的问题

### CombatSystem 过重

旧写法把索敌、移动、攻击队列、Buff、死亡清理、投射物都放进 `CombatSystem`，导致任何战斗波动都会显示为 combat 高耗时。

现在拆分后，combat 行只代表伤害流程。单位移动、索敌、投射物、攻击事件都能单独看。

### 单位模型池导致不可见

整棵 `Object3D.clone(true)` 模型模板会复制旧的 `userData.parts` 和动画引用，容易让单位存在但模型不可见。

当前安全做法：单位模型仍通过工厂创建；以后如果要池化，优先池化 geometry/material 或明确重建 parts 引用，不直接克隆完整运行时根节点。

### 击退后无法恢复移动

被击退时如果走延迟 `requestRepath`，容易卡在冷却或旧路径状态里。

当前规则：击退立刻清掉旧路径；击退和 hit stun 结束后，由单位 AI 自然重新寻路。

### 投射物峰值

投射物 profiler 拆成：

- `projectileFlightMs`：弹体飞行数学计算。
- `projectileMoveApplyMs`：弹体位置/旋转应用。
- `projectileQueryMs`：穿透弹体局部目标查询。
- `projectileHitMs`：命中后的伤害、Buff、飘字、特效、死亡。
- `projectileRecycleMs`：弹体回收和场景移除。

普通箭的飞行不应昂贵。如果看起来贵，优先看是否其实是命中阶段或特效阶段。

### 伤害飘字创建过多

命中阶段频繁创建 canvas 和 `CanvasTexture` 会造成尖峰。

当前 `EffectsSystem` 按文字、颜色、描边、字体等缓存伤害数字纹理，避免每次命中都重新创建。

### 导航和碰撞混在一起

普通单位移动不需要模拟物理碰撞。可不可走由导航网格决定。

当前规则：基地、敌营、小屋、树、墙、柱子等直接写入 nav grid。`isPointWalkable` 只检查边界和网格，不再每帧用结构物碰撞推开单位。

## Profiler 解读

### `单位循环总计`

这是父级汇总。高的时候要继续看子项：

- 单位状态机。
- 移动/寻路。
- 目标决策。
- 攻击决策。
- Buff 更新。
- 支援/建筑影响。

不要只看到这个行高就盲目降频。

### `索敌`

索敌应和局部密度相关，不应随全场单位数量线性暴涨。

如果高：

- 检查是否绕过了 `TargetingSystem.query()`。
- 检查空间网格是否每帧重复构建。
- 检查候选排序是否做了路径判断或复杂规则。

### `移动/寻路`

如果出现周期性尖峰：

- 检查是否目的地没变仍然请求路径。
- 检查追踪目标是否过于频繁重寻路。
- 检查击退是否重复触发落点修正。
- 检查是否把建筑碰撞、基地碰撞等运行时物理判断写回移动循环。

### `投射物`

先看子项：

- 飞行高：检查弹体数量、每帧数学、对象遍历。
- 查询高：检查穿透弹是否扫描全场。
- 命中高：检查伤害流程、Buff 钩子、飘字、命中特效。
- 回收高：检查 scene add/remove、池化和 dispose。

### `渲染`

渲染高不一定要第一时间牺牲画面。先确认战斗逻辑是否仍有尖峰。静态场景阴影可以保留，单位使用假阴影即可。

## 新功能性能检查表

新增单位：

- 是否只在出生/死亡时注册/注销？
- 是否把特殊攻击接入 `AttackSystem`，而不是在 `Game.tick()` 写全局扫描？
- 是否用单位自己的计时器处理技能冷却？

新增 Buff / 附魔：

- 是否挂在单位自己的 Buff 列表？
- tick 是否有明确 interval？
- 是否避免每帧查询全场单位？

新增建筑：

- 建筑 tick 是否有 interval？
- 影响周围单位时是否用空间查询或低频检测？
- 建筑死亡是否主动注销并移除阻挡？

新增法术 / 区域效果：

- 区域持续效果是否低频 tick？
- 命中单位是否用局部查询？
- 特效是否复用材质、纹理或对象池？

新增地图阻挡：

- 是否写进导航网格？
- `N` 可行走调试图是否正确？
- 卡牌落点预览是否跟导航规则一致？

## 验证

每次性能改动后至少运行：

```powershell
node --check <changed-js-file>
npm run build
```

还需要实际验证：

- 单位可见并能移动。
- 近战能攻击。
- 远程能命中。
- 穿透弹不会重复命中同一个单位。
- 击退后单位能恢复移动。
- 死亡单位会注销，不再被索敌或投射物命中。
- 沙漠柱子、雪原树、地牢墙体、基地、敌营、小屋仍不可走。
- Profiler 中 combat 不再承担移动/索敌/投射物父级耗时。
