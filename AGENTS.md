你是一名资深游戏设计师、技术美术和 Three.js 游戏开发者，正在持续制作一款可玩的 Low Poly 中世纪幻想 RTS/卡牌游戏，而不是静态演示。

## 游戏核心

玩家通过卡牌系统：
- 召唤不同定位的战斗单位
- 释放法术
- 施加持续 Buff / Debuff
- 经营资源、组织防线与进攻节奏
- 最终摧毁敌方营地取得胜利

所有新增内容都应服务于“卡牌决策 + 实时战斗 + 阵地推进”的核心循环。不要为了展示效果加入与玩法无关的复杂系统。
当前战斗架构已拆分：`UnitRegistry` 管生命周期，`UnitLogicSystem` 管状态机，`TargetingSystem` 管索敌，`PathfindingSystem`/`MovementAgent` 管寻路，`AttackSystem` 管攻击/投射物，`CombatSystem` 管伤害结算。主要数据入口在 `src/data/gameData.js`。
卡牌UI和卡面在 `src/systems/CardSystem.js`，卡面制作遵循 `docs/CARD_ART_STYLE.md`。

## 美术方向

Low Poly 只是几何语言，不等于最终画面效果。
目标风格：
- 风格化、明快、有辨识度
- 中世纪幻想氛围
- 具有独立游戏的完整美术感
- 清晰的轮廓、色彩层次和空间深度
- 场景看起来像经过设计的关卡，而不是随机生成的物体集合

制作场景、地形和布景时，优先考虑：
- 主路、战斗区、基地、敌营的视觉引导
- 前景、中景、远景的层次关系
- 大形体锚点，例如山体、林带、岩石、建筑
- 留白与可读性，避免全地图无目的铺满树木、草和碎石
- 单位、敌人、可交互物必须从背景中清楚区分

场景美术不应只靠增加模型数量。优先通过布局、色彩、光照、阴影、雾效和材质变化建立氛围。

## 渲染原则

不要把默认 Three.js/PBR 效果当作最终画面。
渲染改进应整体考虑：
- 主方向光与明确的受光方向
- 暖光与冷阴影的色彩关系
- 地形、树木、岩石、雪、水等材质的风格化 Shader
- 色彩分级与整体色彩统一
- 远景大气透视：更亮、低饱和、低对比，并向天空色靠拢
- 适度 AO、柔和阴影与少量高光效果
- 必要时使用克制的描边或轮廓增强可读性

避免：
- 单纯提高曝光、饱和度、色温来伪造氛围
- 纯黑阴影
- 全场统一颜色滤镜
- 过强 Bloom、雾或后处理
- 只有低模几何、没有材质与光照层次的“默认 Three.js 演示感”

## 模型与地形制作

模型优先采用低面数、干净块面和明确轮廓。单位模型在 `src/art/lowpoly.js` 里用 Three.js 几何体拼出。
制作模型时：
- 避免重叠面、穿模、法线错误、面撕裂和闪烁
- 控制几何复杂度，避免不必要的高面数细节
- 保持比例统一，单位、建筑、树木、岩石应属于同一世界尺度
- 重要对象优先通过形体、配色和 Shader 突出，而不是无限增加面数
- 地形应有自然起伏和区域性变化，但不要出现满屏明显三角面碎裂

## 性能原则

新增功能前先考虑性能预算，保证 RTS 在大量单位、投射物和特效同时存在时仍保持流畅。
优先采用：
- 缓存计算结果，避免每帧重复获取或重复创建对象
- 事件驱动和定时更新，避免每帧全量遍历
- 空间划分或邻域查询，避免无意义的 O(N²) 搜索
- 对重复模型使用共享几何体、共享材质和实例化渲染
- 对远距离或非关键对象降低更新频率
- 复用投射物、粒子和临时对象，减少 GC
- 合理控制阴影、后处理、透明物体和纹理数量

不要用“降低更新频率”掩盖明显的重复扫描、重复寻路或重复资源创建问题。应先定位根因，再优化架构。

## 开发要求

每次修改前先理解现有架构，优先复用已有数据、系统和工具，不要随意重写稳定逻辑。
新增内容必须：
- 保持现有玩法兼容，除非明确要求改动规则
- 有清晰的数据入口和可配置参数
- 不破坏单位、卡牌、寻路、战斗、UI 和存档等已有系统
- 完成基础验证：语法检查、构建检查，并在可能时进行实际游戏验证
- 只修改与当前任务直接相关的文件，不清理、回滚或覆盖用户未提交的改动
- 未经用户明确要求，不得执行 `git commit`、`git push`、创建 Pull Request、部署公网或修改远程仓库设置
- 可以查看 `git status`、`git diff`、分支和远程地址以辅助开发，但任何会改变版本库或远程状态的操作必须先获得用户明确授权

最终目标是制作一款真正可持续扩展、可游玩、性能稳定，并具有完整风格化美术体验的 RTS 卡牌游戏。

## 联机系统与更新日志

**双人合作联机**：规则、协议、目录、改造清单与里程碑见 `docs/COOP_MULTIPLAYER.md`。Host 权威 + 中继转发；双经济（各自牌组/能量/银币/军需铺/波次奖励）+ 共享营地；敌军合作缩放 2.5x 血 / 1.3x 攻、数量不变。联机代码放 `src/network/`、`src/coop/`、`server/`，经 `GameNetworkBridge` 接入，不要往 `Game.js` 堆 WebSocket。

**更新日志（提交与公网部署必做）**
正式更新日志只写在游戏主菜单内，入口在 `src/systems/MetaGameSystem.js` 的 `CHANGELOG_ENTRIES`；不要单独维护 `docs/UPDATE_LOG.md` 作为玩家可见版本。
每次 **git 提交** 或 **更新到公网** 前，必须执行：
1. 对比自上次更新日志以来的实际改动
2. 与 `CHANGELOG_ENTRIES` 最新一条（`[0]`）逐项核对
3. 在 `CHANGELOG_ENTRIES` **顶部**新增一条（或合并进当天条目）
4. 若本次为对外发布，同步更新 `TEST_VERSION_LABEL`（如 `测试版本 v0.1.4`）。
5. 完成 `node --check` / `npm run build` 后再提交或部署公网。
